Javascript SDK

  • Handles client-side of OAuth2 flow between HTML5 user application (client) and Keycloak (server)
  • sources in clients/javascript/src/main/javascipt/auth/client.js

Example of JS code in HTML5 page

var liveOak = LiveOak({ auth: { clientId: 'test-app', clientSecret: 'password', onload: 'check-sso', success: authSuccess, error: authError } }); liveOak.auth.init();

Authenticated user

Signed JWT access token available in "window.oauth.init"

Requests to LiveOak sent with Javascript SDK will contain header: Authorization: Bearer <access token>

Request processing in LiveOak

AuthHandler

  • retrieves JWT access token from HTTP header
  • Sends resource request to 'auth' resource with usage of Client to /auth/token-info/<access_token>

Auth resource

After receiving request:

  • verifies signature of access token
  • sends parsed info about authenticated user from token

AuthHandler - continue

establishes SecurityContext and continues with request (handler chain)

SecurityContext

  • Part of LiveOak SPI. Accessible from RequestContext
  • Contract between authentication and authorization

Authz

Powered by Policies

AuthzHandler

  • send authorization request to "authz" resource to verify if current request is authorized
  • Assumption: SecurityContext is already established and filled by AuthHandler (or other handler/interceptor)

Authz resource

delegates decision to one ore more policies

Policies are invoked over REST (async)

A policy can include/exclude specific resources

Policy Decisions

Accept

Reject

Ignore

Accepted if at least one accept and no rejects

Configuration example

"policies": [ { "policyName" : "URIPolicy", "policyResourceEndpoint": "uriPolicy/authzCheck" }, { "policyName" : "ACLPolicy", "policyResourceEndpoint": "aclPolicy/authzCheck", "includedResourcePrefixes": [ "/storage" ] } ]

URIPolicy

Implementation based on Drools

Adding URI rules in configuration, which are internally converted to Drools rules

Allows to authorize based on various characteristics of resourceRequest itself

  • request URI
  • request parameters
  • request body (resourceState)

not just "URIPolicy" as it allows to create conditions based on anything in request

Example of rule

{ "priority": 1, "uriPattern": "/app/(.*)[.](html|css|js|png)", "requestType": "READ", "allowedUsers": "'*'" },

Example 2

{ "priority": 1, "uriPattern": "/storage*", "requestType": "CREATE", "queryParamsCondition": "$requestState.getProperty('user') == $securityContext.subject", "allowedRoles": "'test-app/user'" },

ACLPolicy

Authorize based on ACL attached to resource itself

Using 'Client' to retrieve resource first

Example

Rule:

{ "requestType": "UPDATE", "allowedUserAttribute": "user" }

Resource:

{ "name": "myTodo", "something": "foo", "user": "john" }

This resource can be updated just by user "john"

Example 2

Rule:

{ "requestType": "UPDATE", "allowedUserAttribute": "user", "allowedRolesAttribute": "roles" },

Resource:

{ "name": "todo2", "user": "joe", "roles": ["foo", "bar"] }

Either user "john" or any user with role "foo" or role "bar" is authorized
